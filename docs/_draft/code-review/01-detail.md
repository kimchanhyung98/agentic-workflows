# 프로젝트 코드 리뷰 워크플로우 — 상세 문서

## 개요

대규모 프로젝트의 전체 소스코드를 AI로 리뷰하는 워크플로우이다.
소스코드를 **XML Bundling**으로 구조화하고, [개발 워크플로우](../dev-automation/01-detail.md)의 **멀티 AI 리뷰** 방식으로 병렬 검증한다.

### 핵심 공식

```text
XML Bundling (소스코드 구조화) + 청킹 (컨텍스트 관리) + 멀티 AI 리뷰 (병렬 검증) = 프로젝트 코드 리뷰
```

> 참고: [LLM이 Django에서 보안 취약점을 찾은 방법](https://new-blog.ch4n3.kr/llm-found-security-issues-from-django-ko/)

---

## 아키텍처

[다이어그램 참조](00-diagram.md)

4계층으로 구성된다.

| 레이어              | 역할                            |
|-------------------|-------------------------------|
| Layer 1: Input    | 프로젝트 소스코드, 리뷰 설정              |
| Layer 2: Bundle   | 파일 수집, XML Bundling, 청킹       |
| Layer 3: Review   | 멀티 AI 병렬 리뷰, 결과 종합            |
| Layer 4: Output   | 구조화된 리뷰 리포트                   |

---

## Layer 1: Input (입력)

리뷰 대상 프로젝트와 리뷰 설정을 입력한다.

### 프로젝트 소스코드

리뷰 대상이 되는 프로젝트 디렉토리 전체이다.
Git 저장소를 기준으로, 추적 대상 파일이 수집 범위가 된다.

### 리뷰 설정

리뷰의 범위와 관점을 정의한다.

- **리뷰 관점**: 보안, 코드 품질, 성능, 설계 패턴 등 중점적으로 검토할 영역
- **제외 패턴**: 리뷰에서 제외할 파일/디렉토리 (예: `vendor/`, `*.min.js`, 테스트 픽스처)
- **청킹 전략**: 프로젝트 규모에 따른 분할 방식 (자동 / 모듈 기준 / 수동 지정)
- **대상 언어**: 리뷰할 프로그래밍 언어 필터 (선택)

## Layer 2: Bundle (번들링)

소스코드를 LLM이 이해할 수 있는 구조화된 형식으로 변환한다.

### 워크플로우

```text
디렉토리 순회 → 필터링 → XML Bundling → 컨텍스트 윈도우 확인 → (초과 시) 청킹
```

### 파일 수집

프로젝트 디렉토리를 순회하여 리뷰 대상 파일을 수집한다.

- `.gitignore` 규칙을 기본 제외 패턴으로 적용
- 리뷰 설정의 제외 패턴을 추가 적용
- 바이너리 파일, 미디어 파일 자동 제외
- 빈 파일, 자동 생성 파일(`package-lock.json`, `*.generated.*`) 제외

### XML Bundling

수집된 파일을 하나의 XML 문서로 번들링한다.
파일 경로와 소스코드를 구조화하여 LLM에게 프로젝트의 전체 구조와 파일 간 관계를 제공한다.

#### XML 구조

```xml
<project name="project-name" description="프로젝트 설명">
  <file path="src/models/user.py" language="python">
    <content>
class User(models.Model):
    email = models.EmailField(unique=True)
    password = models.CharField(max_length=128)
    ...
    </content>
  </file>
  <file path="src/views/auth.py" language="python">
    <content>
def login(request):
    email = request.POST.get('email')
    password = request.POST.get('password')
    ...
    </content>
  </file>
  <file path="src/utils/crypto.py" language="python">
    <content>
def hash_password(password):
    ...
    </content>
  </file>
</project>
```

#### XML Bundling의 장점

| 장점          | 설명                                            |
|-------------|-----------------------------------------------|
| 구조 보존       | 파일 경로가 XML 속성으로 포함되어 프로젝트 구조를 유지               |
| 파일 간 관계 파악  | 하나의 문서에 전체 코드가 포함되어 import/참조 관계를 LLM이 직접 추적 가능 |
| 메타데이터 포함    | 파일 경로, 언어 등의 메타데이터를 속성으로 포함                    |
| 청킹 용이       | XML 태그 단위로 분할 가능 — 파일 경계에서 자연스럽게 나눌 수 있음       |
| LLM 친화적     | XML은 LLM이 잘 이해하는 구조화 형식 — 파일 경계와 내용을 명확히 구분    |

#### 일반 텍스트 대비 비교

```text
# 일반 텍스트 연결 방식
=== src/models/user.py ===
(소스코드)
=== src/views/auth.py ===
(소스코드)
```

일반 텍스트 방식은 파일 경계 구분이 모호하고 메타데이터를 표현하기 어렵다.
XML은 태그로 경계를 명확히 하고, 속성으로 메타데이터를 자연스럽게 포함한다.

### 청킹

XML Bundle이 모델의 컨텍스트 윈도우를 초과하면 분할한다.

#### 청킹 전략

| 전략     | 설명                                | 적합한 경우                  |
|--------|-----------------------------------|-------------------------|
| 단일 청크  | 분할 없이 전체 프로젝트를 한 번에 전달            | 소규모 프로젝트 (컨텍스트 윈도우 이내)  |
| 모듈 기준  | 디렉토리/패키지 단위로 분할                   | 모듈화된 프로젝트               |
| 의존성 기준 | import/참조 관계가 있는 파일을 같은 청크에 배치    | 파일 간 의존성이 중요한 리뷰        |
| 크기 기준  | 토큰 수를 기준으로 균등 분할                  | 대규모 프로젝트, 빠른 분할이 필요한 경우 |

#### 청킹 시 주의사항

- 관련 파일은 같은 청크에 배치한다 — 모델이 파일 간 관계를 파악할 수 있어야 한다
- 공통 유틸리티, 설정 파일은 모든 청크에 포함하거나 별도 컨텍스트로 제공한다
- 각 청크에 프로젝트 구조 요약을 포함하여 전체 맥락을 보완한다

## Layer 3: Review (리뷰)

[개발 워크플로우의 멀티 AI 리뷰](../dev-automation/01-detail.md#멀티-ai-리뷰-구조)와 동일한 구조를 적용한다.
각 청크를 복수의 AI 모델이 병렬로 검증한다.

### 워크플로우

```text
청크 전달 → 병렬 리뷰 (복수 AI) → 리뷰 문서 생성 → 종합 · 중복 제거 → 리뷰 리포트
```

### 리뷰 프롬프트 구성

각 리뷰어에게 전달하는 프롬프트는 다음 요소로 구성된다.

- **리뷰 관점**: 리뷰 설정에서 정의한 중점 검토 영역
- **XML Bundle 또는 청크**: 구조화된 소스코드
- **프로젝트 구조 요약**: 전체 디렉토리 구조와 파일 목록 (청크 분할 시 필수)
- **출력 형식 지시**: 구조화된 JSON 또는 마크다운 형식으로 리뷰 결과 출력

### 리뷰어 구성

각 리뷰어는 서로 다른 관점에서 코드를 검증한다.

| 역할    | 검증 관점                                                  |
|-------|--------------------------------------------------------|
| 리뷰어 1 | **보안**: 인증/인가, 입력 검증, SQL 인젝션, XSS, 민감정보 노출, 암호화      |
| 리뷰어 2 | **코드 품질**: 설계 패턴, 코드 중복, 네이밍, 에러 처리, 유지보수성, SOLID 원칙  |
| 리뷰어 3 | **성능**: N+1 쿼리, 메모리 누수, 불필요한 연산, 캐싱, 리소스 관리, 동시성 이슈   |

리뷰어 구성은 프로젝트 특성에 따라 조정한다.
예: 프론트엔드 프로젝트는 접근성·UX 관점을 추가하고, 데이터 파이프라인은 데이터 정합성 관점을 추가한다.

### 멀티 AI 리뷰 흐름

1. XML Bundle(또는 분할된 청크)을 각 리뷰어에게 병렬로 전달
2. 각 리뷰어가 자신의 관점에서 독립적으로 리뷰 문서를 생성
3. 오케스트레이션 AI가 모든 리뷰 문서를 종합
4. 중복 이슈 제거 및 심각도 재판정
5. 최종 리뷰 리포트를 생성

### 복수 청크 처리

프로젝트가 여러 청크로 분할된 경우:

- 각 청크에 대해 독립적으로 멀티 AI 리뷰를 수행
- 모든 청크의 리뷰 결과를 최종 종합 단계에서 합산
- 청크 간 걸쳐 있는 이슈(크로스 모듈 의존성 등)는 종합 단계에서 식별

## Layer 4: Output (산출물)

### 리뷰 리포트

구조화된 리뷰 리포트를 출력한다.

#### 리포트 구성

- **요약**: 전체 리뷰 결과 개요 — 이슈 수, 심각도 분포
- **이슈 목록**: 파일별 이슈 — 위치, 심각도, 설명, 개선 제안
- **주요 발견**: 프로젝트 전반에 걸친 패턴 이슈 또는 구조적 문제
- **긍정적 측면**: 잘 작성된 코드, 좋은 패턴 활용 사례

#### 심각도 분류

| 심각도   | 설명                         | 예시                  |
|-------|----------------------------|---------------------|
| 심각    | 즉시 수정 필요 — 보안 취약점, 데이터 손실 위험 | SQL 인젝션, 하드코딩된 비밀키  |
| 높음    | 빠른 수정 권장 — 버그, 심각한 설계 결함     | 인증 우회 가능성, 메모리 누수   |
| 중간    | 개선 권장 — 코드 품질, 유지보수성 이슈       | 코드 중복, 부적절한 에러 처리   |
| 낮음    | 참고 사항 — 스타일, 사소한 개선 가능 사항     | 네이밍 컨벤션, 주석 부족      |

#### 이슈 출력 형식 예시

```text
[심각] src/views/auth.py:42
  SQL 인젝션 취약점
  - f-string으로 사용자 입력을 직접 쿼리에 삽입
  - 개선: ORM 또는 파라미터화된 쿼리 사용

[높음] src/utils/crypto.py:15
  약한 해시 알고리즘 사용 (MD5)
  - 비밀번호 해싱에 MD5 사용 — 충돌 공격에 취약
  - 개선: bcrypt 또는 argon2 사용

[중간] src/models/user.py:28
  에러 처리 누락
  - 데이터베이스 조회 실패 시 예외 처리 없음
  - 개선: try-except 블록으로 에러 처리 추가
```

## XML Bundling 상세

### 번들링 프로세스

```text
1. 프로젝트 루트 디렉토리 지정
2. .gitignore + 제외 패턴으로 필터링 규칙 생성
3. 디렉토리 재귀 순회 → 대상 파일 목록 수집
4. 각 파일의 메타데이터(경로, 언어) 추출
5. XML 구조로 변환
6. 토큰 수 계산 → 컨텍스트 윈도우 초과 시 청킹
```

### XML 태그 설명

| 태그          | 속성                  | 설명                   |
|-------------|---------------------|----------------------|
| `<project>` | `name`, `description` | 프로젝트 루트 — 이름과 설명 포함  |
| `<file>`    | `path`, `language`    | 개별 파일 — 경로와 언어 정보 포함 |
| `<content>` | —                   | 파일의 실제 소스코드 내용       |

### 대규모 프로젝트 적용

대규모 프로젝트(수천 개 파일)의 경우 다음을 고려한다.

- **우선순위 기반 수집**: 핵심 비즈니스 로직부터 번들링 — 테스트, 설정 파일은 후순위
- **점진적 리뷰**: 모듈 단위로 나누어 순차적으로 리뷰
- **디렉토리 구조 요약**: 전체 파일 목록을 별도 컨텍스트로 제공하여 각 청크에 전체 맥락 부여
- **프로젝트 구조 맵 포함**: 각 청크 앞에 `<structure>` 태그로 전체 디렉토리 트리 포함

```xml
<project name="large-project">
  <structure>
src/
├── models/
│   ├── user.py
│   └── order.py
├── views/
│   ├── auth.py
│   └── order.py
└── utils/
    └── crypto.py
  </structure>
  <file path="src/models/user.py" language="python">
    <content>...</content>
  </file>
  <!-- 이 청크에 포함된 파일들 -->
</project>
```

## 멀티 AI 리뷰와 개발 워크플로우 연계

이 코드 리뷰 워크플로우는 [개발 워크플로우](../dev-automation/01-detail.md)의 멀티 AI 리뷰 구조를 재사용한다.

### 공통 구조

| 요소          | 개발 워크플로우              | 코드 리뷰 워크플로우       |
|-------------|------------------------|-------------------|
| 리뷰 대상      | 기획 문서 또는 코드 diff       | 프로젝트 전체 소스코드      |
| 입력 형식      | 기획 문서 또는 diff           | XML Bundle        |
| 병렬 리뷰      | 복수 AI 모델이 독립적으로 검증     | 동일                |
| 결과 종합      | 오케스트레이션 AI가 판정         | 동일                |
| 피드백 루프     | 실패 시 수정 → 재리뷰          | 없음 (일회성 리뷰)       |

### 차이점

- 개발 워크플로우는 **코드 변경(diff) 기준**으로 리뷰하지만, 코드 리뷰 워크플로우는 **프로젝트 전체**를 리뷰한다
- 개발 워크플로우는 기획 문서 대조가 핵심이지만, 코드 리뷰 워크플로우는 **일반적인 코드 품질 관점**으로 리뷰한다
- 코드 리뷰 워크플로우에는 자동 수정 루프가 없다 — 리포트를 산출물로 제공한다

## 적용된 설계 패턴

| 패턴              | 적용 위치                | 설명                            |
|-----------------|----------------------|-------------------------------|
| Parallel        | 멀티 AI 리뷰             | 복수 모델이 병렬로 검증, 결과 집계          |
| MapReduce       | 청크별 리뷰 → 종합          | 분할된 청크를 독립 리뷰 후 결과를 합산        |
| Sectioning      | 청킹                   | 대규모 입력을 독립적인 섹션으로 분할하여 병렬 처리  |
| Review-Critique | 멀티 AI 리뷰             | 각 리뷰어가 서로 다른 관점에서 비평          |

## 검토 사항

- XML Bundling 도구는 CLI 스크립트로 제공할 것인가, 에이전트가 직접 수행할 것인가?
- 청킹 시 관련 파일 그룹화의 기준은? (디렉토리, import 관계, 수동 지정)
- 리뷰어 모델 구성은 고정인가, 프로젝트별 설정인가?
- 리뷰 결과의 출력 형식은? (마크다운, JSON, GitHub Issue)
- 정기적 리뷰(cron)와 수동 실행을 모두 지원할 것인가?
- 리뷰 결과를 개발 워크플로우의 기획 단계에 피드백으로 활용할 수 있는가?
