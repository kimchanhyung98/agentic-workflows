# 로컬 개발 에이전트 워크플로우 — 상세 문서

## 개요

Stripe Minions의 아키텍처를 로컬 환경에 맞게 변형한 개발 자동화 에이전트이다.
AI CLI를 에이전트 코어로 활용하고, Minions의 One-Shot 방식 대신 **실행 전 사람의 리뷰를 한 번 거치는** 구조를 취한다.
검증은 단일 AI가 아닌 **멀티 AI 리뷰**(병렬 패턴)로 수행한다.

### 핵심 공식

```text
AI CLI (기존 도구 활용) + 멀티 AI 리뷰 + 하네스 (hook · 게이트 · 기획 문서) + 사람 리뷰 = 로컬 개발 자동화
```

> Minions: "Not exotic — just great engineering"
> 로컬 에이전트: 에이전트를 만드는 것이 아니라, **에이전트가 동작하는 환경을 설계**한다.

---

## 아키텍처

[다이어그램 참조](00-diagram.md)

4계층 + Human Review로 구성된다.

| 레이어                         | 역할                            | Minions 대응                  |
|-----------------------------|-------------------------------|-----------------------------|
| Layer 1: Entry Point        | 작업 진입 (CLI, Web UI, OpenClaw) | Entry Points (Slack, CLI 등) |
| Layer 2: Prepare            | 컨텍스트 수집, 기획 문서 작성, 멀티 AI 리뷰   | Context Hydration (MCP)     |
| Layer 3: Execute + Feedback | 코드 작성, 정적 분석, 멀티 AI 리뷰, 자동 수정 | Agent Core + Feedback Loop  |
| Layer 4: Output             | Commit / PR 결과물 정리            | Output (PR)                 |
| Human Review                | 기획 리뷰, 자동 수정 실패 시 실패 리포트      | — (Minions에 없음)             |

## Layer 1: Entry Point

작업은 여러 경로로 Main AI CLI에 전달된다.

| 채널        | 설명                                                                   |
|-----------|----------------------------------------------------------------------|
| CLI 직접 실행 | 개발자가 터미널에서 AI CLI를 직접 실행                                             |
| Web UI    | 웹 인터페이스를 통해 요청 → 로컬 서버 → AI CLI에 전달                                  |
| OpenClaw  | Slack, Discord 등 메신저 봇을 통해 요청 → AI CLI에 전달. 메신저와 AI CLI를 연결하는 브릿지 역할 |

모든 경로는 최종적으로 Main AI CLI(오케스트레이터)에 도달한다.
Main AI CLI가 이후의 모든 단계를 총괄한다 — 별도의 오케스트레이터를 구축하지 않는다.

### Minions와의 비교

Minions는 4가지 Entry Point(Slack, CLI, Web UI, CI Auto-Ticket)에서 Orchestrator로 작업이 전달된다.
로컬 에이전트는 AI CLI가 Orchestrator 역할을 겸한다.

## Layer 2: Prepare (준비 단계)

에이전트가 요청을 받아 컨텍스트를 수집하고, 기획 문서를 작성한다.
기획 문서는 멀티 AI 리뷰를 거친 후 사람에게 공유된다.

### 워크플로우

준비 단계는 두 단계로 구성된다: **조사 + 기획 문서 작성**, **멀티 AI 리뷰**.

```text
요구사항 분석 + 프로젝트 탐색 (Augment Context MCP)
    → (필요 시) 추가 조사
    → (존재 시) AI 컨텍스트 로딩
    → 기획 문서 작성 (기획안 + 테스트 케이스 개요)
    → 멀티 AI 리뷰
    → (부족) 요구사항 분석으로 되돌아감
    → (충분) 사람에게 기획 문서 공유
    → 승인 / 수정 요청
```

### 각 단계 설명

#### 요구사항 분석 + 프로젝트 탐색

Augment Context MCP를 활용하여 요청을 분석하고 프로젝트 내부 정보를 파악한다.

- 요청의 의도와 범위 파악, 암묵적 요구사항 식별
- 시맨틱 검색으로 관련 파일과 코드 관계 파악
- 프로젝트 구조와 기존 패턴 파악
- Git 이력에서 관련 변경 확인

Augment Context MCP는 키워드 불일치 시에도 의미적으로 관련된 코드를 발견할 수 있다.
기존의 정적 분석(grep, glob, import 추적)을 시맨틱 검색으로 보완하는 역할이다.

#### 추가 조사 (선택 — AI 판단)

프로젝트 외부의 정보가 필요한 경우에만 수행한다.
AI가 요구사항 분석 결과를 바탕으로 추가 조사 필요 여부를 판단한다.

- 유사한 작업의 주의사항, 엣지케이스 조사
- 관련 라이브러리/API의 최신 사용법 확인
- 알려진 문제점이나 함정 파악

단순한 작업은 추가 조사 없이 바로 기획 문서 작성으로 진행할 수 있다.

#### AI 컨텍스트 로딩 (선택 — 존재 시)

프로젝트에 사전 구조화된 AI 컨텍스트 파일이 존재하는 경우 로딩한다.
도메인 개요, 데이터 모델, API 스펙 등 아키텍처 레이어에 매핑된 정적 지식이다.

- 도메인 정책과 유스케이스 파악
- 엔티티 정의와 데이터 모델 확인
- API 스펙, 이벤트 스펙 등 인터페이스 파악

AI 컨텍스트가 없는 프로젝트에서는 이 단계를 건너뛴다.
Augment Context MCP의 시맨틱 검색이 동적으로 컨텍스트를 수집하므로, AI 컨텍스트는 필수가 아닌 보완 수단이다.
사전에 잘 정리된 컨텍스트가 있으면 탐색 정확도와 토큰 효율이 올라간다.

#### 기획 문서 작성

기획안과 테스트 케이스 개요를 하나의 기획 문서로 정리한다.

- 실행 단계의 입력이 되는 문서 — 에이전트는 이 문서를 기준으로 구현
- 사람에게 공유하여 리뷰를 받는 용도도 겸함
- 자세하고 논리적인 기획안 작성이 핵심 — 기획의 품질이 구현의 품질을 결정

**기획 문서에 포함되는 것:**

- **컨텍스트**: 식별된 관련 파일 목록
- **AI 컨텍스트**: 로딩한 도메인·데이터 모델·API 스펙 요약 (존재하는 경우)
- **추가 조사 결과**: 주의사항, 엣지케이스 (조사한 경우)
- **기획안**: 무엇을 왜 변경하는지에 대한 자세하고 논리적인 설명
- **작업 계획**: 어떤 파일을 어떻게 수정할지
- **테스트 케이스 개요**: 핵심 시나리오 수준의 간단한 검증 항목
- **애매한 판단**: 에이전트가 확신하지 못하는 부분 (사람의 결정 필요)

#### 멀티 AI 리뷰

기획 문서를 복수의 AI 모델이 병렬로 검증한다.
각 AI가 리뷰 문서를 생성하고, 오케스트레이션 AI가 결과를 종합하여 판정한다.

- 부족하면 → 요구사항 분석으로 되돌아감
- 충분하면 → 사람에게 기획 문서 공유

단일 AI의 판단에 의존하지 않고, 복수 모델의 교차 검증으로 기획 품질을 확보한다.
멀티 AI 리뷰의 상세 구조는 [멀티 AI 리뷰 섹션](#멀티-ai-리뷰-구조)을 참조한다.

#### Human Review (기획 문서 리뷰)

멀티 AI 리뷰를 통과한 기획 문서를 사람에게 공유한다.

- 기획안의 방향과 범위가 적절한지 확인
- 테스트 케이스 개요가 핵심 시나리오를 커버하는지 확인
- 애매한 판단 항목에 대해 결정
- 승인 시 실행 단계로 진행, 수정 요청 시 요구사항 분석으로 되돌아감

이것이 Minions와의 핵심 차이이다.
Minions는 One-Shot으로 사람의 개입 없이 PR까지 완료하지만, 로컬 에이전트는 **실행 전 사람의 리뷰를 한 번 거친다**.
이 리뷰가 로컬 에이전트의 권한 모델이 된다 — **리뷰가 곧 권한**.

### Minions Layer 2 대응

| Minions                        | 로컬 에이전트                      |
|--------------------------------|------------------------------|
| Toolshed MCP (400+ 도구)         | Augment Context MCP (시맨틱 검색) |
| 결정론적 pre-fetching              | 도구 기반 결정론적 탐색 + 시맨틱 검색       |
| Link Resolver + Prompt Scanner | 요구사항 분석                      |
| Context Builder                | 기획 문서 작성                     |
| —                              | 멀티 AI 리뷰 (Minions에 없음)       |

## Layer 3: Execute + Feedback (실행 단계)

사람이 기획 문서를 승인하면, 에이전트가 기획 문서를 기준으로 자율적으로 구현한다.
실행 환경은 로컬 또는 VM/Docker에서 선택한다.

### 실행 환경

실행 환경은 프로젝트 설정 또는 실행 시 옵션으로 선택한다.

| 환경        | 격리 수준 | 안전성 모델              | 적합한 경우         |
|-----------|-------|---------------------|----------------|
| 로컬        | 없음    | 사람의 사전 검토 + 게이트     | 개인 프로젝트, 빠른 실행 |
| VM/Docker | 높음    | 환경 격리 + 사전 검토 + 게이트 | 팀 프로젝트, 높은 안전성 |

`--auto` 모드: VM/Docker 격리 환경과 조합하면, 사람의 리뷰 없이 One-Shot으로 실행할 수 있다.
다만 안전성은 격리 자체가 아니라, 프로덕션 자격증명 차단, 네트워크 격리, 임시 실행 환경 등의 전제 조건이 함께 적용될 때 확보된다.

### 워크플로우

코드 작성과 정적 분석(test, lint 등)은 하나의 단위로 동작한다.
코드를 작성하면 당연히 테스트와 lint를 실행하고, 실패 시 즉시 코드를 수정한다.

```text
기획 문서 확인 → 코드 작성 (Main AI)
    → 정적 분석 (test, lint 등) → 실패 시 코드 수정 후 재실행
    → 멀티 AI 리뷰 → 실패 시 자동 수정 (최대 N회) → 재시도
    → 통과 시 Output (Commit / PR)
    → 한도 초과 시 실패 리포트 작성
```

### 각 단계 설명

#### 코드 작성 + 정적 분석

기획 문서를 기반으로 코드를 생성·수정한다.
정적 분석(test, lint 등)은 코드 작성의 일부로 자동 실행된다.

- 기획안, 작업 계획, 테스트 케이스 개요를 기준으로 코드 작성
- `make check` 등의 hook으로 정적 분석 자동 실행
- lint, type check, test 등 결정론적 검증
- 실패 시 즉시 코드를 수정하고 재실행 — 이 루프는 재시도 횟수에 포함하지 않음

정적 분석은 LLM이 우회할 수 없는 하드코딩된 게이트이다.
Minions의 핵심 원칙: 게이트는 프롬프트가 아닌 **구조**로 강제한다.

#### 멀티 AI 리뷰

정적 분석을 통과한 코드를 복수의 AI 모델이 병렬로 검증한다.

- diff를 기반으로 실제 변경 내용을 파악
- 기획 문서의 기획안과 실제 구현을 비교하여 요구사항 누락, 범위 이탈 확인
- 추가 조사 단계에서 파악한 주의사항·엣지케이스가 반영되었는지 점검
- 정적 분석으로 잡을 수 없는 논리 오류, 경계 조건, 사이드 이펙트 검토
- 준비 단계에서 작성한 테스트 케이스 개요가 커버되는지 확인

각 AI가 리뷰 문서를 생성하고, 오케스트레이션 AI가 결과를 종합하여 통과/실패를 판정한다.
실패 시 구조화된 피드백을 Main AI에 전달한다.

#### 자동 수정 (최대 N회)

멀티 AI 리뷰에서 실패하면, 구조화된 피드백을 기반으로 코드를 수정한다.

- 자동 수정 횟수를 제한한다 — 무한 루프 방지
- 한도를 초과하면 실패 리포트를 작성하여 사람에게 전달

Minions의 원칙: "Can't fix in 2 tries? Surface it to the human — no wasted tokens"

### Minions Agent Core + Feedback Loop 대응

Minions는 Agent Core와 Feedback Loop를 별도 레이어로 분리하지만,
로컬 에이전트는 하나의 실행 레이어로 통합한다.

| 구분      | Minions                                          | 로컬 에이전트                    |
|---------|--------------------------------------------------|----------------------------|
| LLM 단계  | Think (LLM) → Write (LLM)                        | 기획 문서 확인 → 코드 작성 (Main AI) |
| Gate 단계 | Linter (Gate) → Git Commit (Gate) → Tests (Gate) | 정적 분석 (test, lint 등) — 통합  |
| LLM 검증  | Review (LLM) — 단일 AI                             | 멀티 AI 리뷰 — 복수 모델 교차 검증     |
| 자동 수정   | Agent Self-Fix (max 2)                           | 자동 수정 (최대 N회)              |
| 실패 시    | 사람에게 전달                                          | 실패 리포트 작성                  |

## Human Review

사람의 개입이 필요한 두 가지 시점이 있다.

### 기획 문서 리뷰

준비 단계에서 멀티 AI 리뷰를 통과한 기획 문서를 사람이 검토한다.

- 승인 시 실행 단계로 진행
- 수정 요청 시 요구사항 분석으로 되돌아감

이 리뷰는 Human-in-the-Loop 패턴의 적용이다.
에이전트 실행의 체크포인트에서 일시 정지하고, 사람의 판단을 받은 후 진행한다.

### 자동 수정 실패 시

실행 단계에서 자동 수정이 한도를 초과하면 실패 리포트를 작성한다.

- 멀티 AI 리뷰의 구조화된 피드백을 기반으로 실패 원인을 정리
- 사람이 실패 리포트를 확인하고 문제를 해결하거나, 기획을 수정하여 다시 시도

---

## Layer 4: Output (산출물)

모든 레이어를 통과한 결과물은 Commit 또는 PR이다.

### 산출물 완성 기준

- 정적 분석(test, lint 등) 통과
- 멀티 AI 리뷰에서 기획 충족 확인
- 기획 문서의 테스트 케이스 개요 충족

### PR 산출물에 포함되는 것

- 변경 사항 요약 — 기획 문서의 기획안을 기반으로 작성
- 기획 문서 참조 — 원본 기획 문서에 대한 링크 또는 인용
- 테스트 결과 — 정적 분석 및 멀티 AI 리뷰 통과 여부
- 요청자에게 완료 알림 (Entry Point 채널에 따라 CLI 출력, Web UI 알림, 메신저 알림)

## 멀티 AI 리뷰 구조

[다이어그램 참조](00-diagram.md#멀티-ai-리뷰)

멀티 AI 리뷰는 준비 단계(기획 문서 검증)와 실행 단계(코드 검증) 모두에서 동일한 구조로 사용된다.
병렬 패턴을 적용하여 복수의 AI 모델이 독립적으로 검증한다.

### 구성

| 역할         | 모델 예시         | 검증 관점              |
|------------|---------------|--------------------|
| 리뷰어 1      | Claude Opus   | 기획 대조, 논리 검증       |
| 리뷰어 2      | GPT 5.4       | 품질, 엣지케이스          |
| 리뷰어 3      | GPT 5.3 Codex | 코드 정합성, 버그 탐지      |
| 오케스트레이션 AI | —             | 리뷰 문서 종합, 통과/실패 판정 |

### 흐름

1. 산출물(기획 문서 또는 코드)을 각 리뷰어에게 병렬로 전달
2. 각 리뷰어가 독립적으로 리뷰 문서를 생성
3. 오케스트레이션 AI가 리뷰 문서를 종합하여 통과/실패 판정
4. 실패 시 구조화된 피드백을 Main AI에 전달하여 수정

### 병렬 패턴 적용 이유

- 모든 리뷰어가 동시에 실행 → 지연 시간 최소화
- 각 리뷰어가 다른 관점에서 검증 → 단일 AI의 사각지대 보완
- 결과 집계가 결정론적 → 일관된 판정 기준

## 설계 원칙

### 기존 도구 활용

에이전트 전용 도구를 만들지 않는다.
개발자가 이미 사용하는 AI CLI, lint, test, Git을 그대로 활용한다.

Minions의 원칙: "Human Engineers = AI Agents" — 에이전트도 사람과 동일한 도구를 사용한다.

### 하네스를 설계하라

> "Design the harness, not the agent"

에이전트(AI CLI) 자체를 개선하는 것이 아니라, 에이전트가 동작하는 시스템 환경에 투자한다.
hook, 게이트, 기획 문서 구조, 실행 환경 격리 등이 하네스에 해당한다.

### LLM 창의성 + 결정론적 신뢰성

코드 생성은 LLM에게, 검증은 하드코딩된 게이트에 맡긴다.
게이트는 프롬프트가 아닌 구조로 강제한다 — LLM이 우회할 수 없다.

| 유형   | 단계                 | 특성                  |
|------|--------------------|---------------------|
| LLM  | 코드 작성              | 창의적 판단, 유연한 문제 해결   |
| Gate | 정적 분석 (test, lint) | 결정론적, 항상 실행, 우회 불가  |
| LLM  | 멀티 AI 리뷰           | 기획 대조, 논리 검증, 맥락 이해 |
| Gate | Commit / PR        | 결정론적, 검증 통과 후에만 실행  |

### 멀티 AI 리뷰

단일 AI의 판단에 의존하지 않고, 복수 모델의 교차 검증으로 품질을 확보한다.
기획 문서 검증과 코드 검증 모두에 동일한 병렬 리뷰 구조를 적용한다.

### 결정론적 컨텍스트 수집

LLM의 판단에 의존하지 않는 도구 기반 탐색으로 일관된 품질을 확보한다.
Augment Context MCP의 시맨틱 검색으로 정적 분석의 의미적 한계를 보완한다.

### 제한된 재시도

자동 수정 횟수를 제한한다. 해결할 수 없으면 실패 리포트를 작성한다.
무한 재시도로 토큰을 낭비하지 않는다.

Minions: "Can't fix in 2 tries? Surface it to the human — no wasted tokens"

## 적용된 설계 패턴

| 패턴                   | 적용 위치              | 설명                             |
|----------------------|--------------------|--------------------------------|
| Parallel             | 멀티 AI 리뷰 (준비 + 실행) | 복수 모델이 병렬로 검증, 결과 집계           |
| Prompt Chaining      | 준비 단계 전체           | 순차적 분해: 요구사항 분석 → 조사 → 기획 → 문서 |
| Evaluator-Optimizer  | 멀티 AI 리뷰 루프        | 산출물 품질을 평가하고 기준 미달 시 수정 반복     |
| Review-Critique      | 멀티 AI 리뷰           | 작성(Generator)과 검증(Critic) 분리   |
| Human-in-the-Loop    | 기획 리뷰, 자동 수정 실패 시  | 체크포인트에서 사람의 판단을 받고 진행          |
| Iterative Refinement | 실행 단계 자동 수정 루프     | 코드 → 검증 → 수정을 반복하여 품질 개선       |

## Minions와의 전체 비교

|             | Stripe Minions                      | 로컬 에이전트                          |
|-------------|-------------------------------------|----------------------------------|
| 실행 방식       | One-Shot (PR까지 자율 완료)               | 기획 리뷰 후 실행                       |
| 사람 개입       | 없음 (PR 리뷰만)                         | 실행 전 1회 리뷰 + 실패 시 리포트            |
| 안전성 모델      | VM 격리 (격리가 곧 권한)                    | 사람의 사전 검토 + 게이트 (리뷰가 곧 권한)       |
| 에이전트 코어     | Goose Fork (커스텀)                    | AI CLI (기존 도구 활용)                |
| 컨텍스트 수집     | Toolshed MCP (400+ → 15개)           | Augment Context MCP (시맨틱 검색)     |
| 코드 검증       | Review (LLM) — 단일 AI                | 멀티 AI 리뷰 — 복수 모델 교차 검증           |
| 실행 환경       | Devbox (VM only)                    | 로컬 / VM / Docker (선택)            |
| 피드백 루프      | 3-Tier (Lint → CI → Self-Fix max 2) | 정적 분석 → 멀티 AI 리뷰 → 자동 수정 (max N) |
| 실패 처리       | 사람에게 전달                             | 실패 리포트 작성                        |
| 규모          | 대규모 모노레포 (주당 1,000+ PR)             | 개인/소규모 프로젝트                      |
| Entry Point | Slack, CLI, Web UI, CI Auto-Ticket  | CLI, Web UI, OpenClaw            |

## 검토 사항

- 기획 문서 형식과 제공 방식은? (터미널 출력, 마크다운 파일, IDE)
- 리뷰 없이 바로 실행하는 옵션(`--auto`)이 필요한가? VM/Docker 격리와 조합하면 가능
- VM/Docker 실행 시 환경 구성은? (Dockerfile, docker-compose)
- Git 브랜치 분리 vs Worktree?
- 자동 수정 최대 횟수는? (Minions는 2회)
- 어떤 AI CLI를 기본으로 지원할 것인가?
- 멀티 AI 리뷰의 리뷰어 모델 구성은 고정인가, 프로젝트별 설정인가?
- 리뷰 문서 수집 주기와 타임아웃 정책은?
