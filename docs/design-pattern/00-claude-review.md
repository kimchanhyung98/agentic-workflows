# 문서 리뷰 결과

`docs/design-pattern/` 디렉토리의 문서를 [Google Cloud - Choose a design pattern for an agentic AI system](https://cloud.google.com/architecture/choose-design-pattern-agentic-ai-system) 원문과 대조하여 검토한 결과입니다.

---

## Critical Issues

### 1. 단일 에이전트의 "단일 추론-행동 루프" 표현 오류

- **위치**: `01-single-agent.md` 11행
- **현재**: "단일 추론-행동 루프로 작업 완수"
- **문제**: 단일 에이전트도 여러 번의 도구 호출과 추론 단계를 반복할 수 있음. "단일 에이전트"를 "단일 반복"으로 오해하게 만드는 표현. 같은 문서의 시퀀스 다이어그램(30-42행)에서 `alt 추가 도구 필요` 분기를 보여주고 있어 본문과 다이어그램이 모순됨.
- **원문 근거**: Google Cloud 문서의 단일 에이전트 정의에는 "단일 루프"라는 제약이 없으며, "에이전트는 모델의 추론 기능으로 사용자 요청을 해석하고, 단계별 계획을 수립하며, 정의된 도구 집합에서 사용할 도구를 결정합니다"라고 반복적 추론을 시사함.
- **수정 제안**: "추론과 도구 호출의 반복을 통해 작업 완수"

### 2. 단일 에이전트의 비용 설명 부정확

- **위치**: `01-single-agent.md` 장단점 표
- **현재**: "단일 모델 호출로 비용 효율적"
- **문제**: 단일 에이전트도 여러 번의 모델 호출이 발생함 (각 도구 호출마다 추론 필요). "단일 호출"이라는 표현은 사실과 다름.
- **원문 근거**: 원문은 비용 효율성을 "단일 호출" 때문이 아니라 멀티 에이전트 대비 오케스트레이션 오버헤드가 없기 때문으로 설명. ReAct 섹션에서 "단일 ReAct 에이전트는 복잡한 멀티 에이전트 시스템보다 구현하고 유지하는 것이 더 간단하고 비용 효율적일 수 있습니다"라고 명시.
- **수정 제안**: "오케스트레이션 오버헤드 없이 멀티 에이전트 대비 비용 효율적"

### 3. 스웜 패턴의 디스패처 역할 설명 모호

- **위치**: `09-swarm.md` 8-10행
- **현재**: "디스패처 에이전트가 요청을 라우팅하고 사용자 통신 지원" + "중앙 코디네이터 없는 분산형 협업"
- **문제**: 두 문장이 함께 있으면 "디스패처가 곧 코디네이터 아닌가?"라는 혼란 발생. 디스패처의 제한적 역할과 코디네이터와의 차이가 불명확.
- **원문 근거**: 원문은 "디스패처 에이전트는 에이전트 워크플로를 오케스트레이션하지 않습니다. 대신 디스패처 에이전트는 스웜 하위 에이전트와 사용자 간의 커뮤니케이션을 지원합니다"라고 명시적으로 구분.
- **수정 제안**: "디스패처는 초기 요청 전달과 사용자 응답 전달만 담당하며, 에이전트 간 협업 과정을 지시하거나 제어하지 않음 (코디네이터와의 핵심 차이)"라는 설명 추가

### 4. ReAct 패턴 장점의 번역 누락

- **위치**: `10-react.md` 장단점 표
- **현재**: "단순한 멀티 에이전트 시스템보다 구현/유지보수 간단"
- **문제**: 원문의 "complex multi-agent systems"를 "단순한 멀티 에이전트"로 잘못 번역. "비용 효율적(cost-effective)" 부분도 누락됨. "단순한"이 수식하는 대상이 바뀌어 의미가 변질됨.
- **원문 근거**: "단일 ReAct 에이전트는 **복잡한** 멀티 에이전트 시스템보다 구현하고 유지하는 것이 더 간단하고 **비용 효율적**일 수 있습니다"
- **수정 제안**: "복잡한 멀티 에이전트 시스템보다 구현, 유지보수, 비용 면에서 효율적"

## Important Issues

### 5. README 복잡도/비용 비교표 정확성

- **위치**: `README.md` 복잡도 및 비용 비교 표
- **현재**: 단일 에이전트 지연 시간 "짧음", 루프 패턴 복잡도 "낮음"
- **문제**:
  - 단일 에이전트의 지연 시간은 도구 호출 횟수에 따라 변동. 원문에서도 단점으로 "지연 시간 증가"를 언급.
  - 루프 패턴은 사전 정의된 로직이지만 무한 루프 위험과 예측 불가능한 반복 횟수가 있어 "낮음"보다 "변동"이 적합. 원문에서 "운영 비용이 과도하게 발생하고 리소스 소비가 많아지며 시스템이 멈출 수 있습니다"라고 경고.
- **수정 제안**: 단일 에이전트 지연 시간을 "변동", 루프 패턴 비용/지연 시간을 "변동"으로 변경

### 6. README 선택 가이드 flowchart의 반복 개선 패턴 중복

- **위치**: `README.md` 패턴 선택 가이드 flowchart
- **현재**: 반복 개선 패턴이 "결정론적 워크플로"와 "반복 포함 워크플로" 두 곳에 동시 등장
- **문제**: 동일 패턴이 두 카테고리에 표시되어 독자 혼란 가능.
- **원문 근거**: Google Cloud 원문에서도 반복 개선 패턴을 "결정론적 워크플로"와 "반복이 포함된 워크플로" 양쪽 표에 모두 포함시키고 있음. 즉, 원문에 충실한 표현이나 주석이 없어 혼란 발생 가능.
- **수정 제안**: flowchart에 주석 또는 하단 설명 추가: "반복 개선 패턴은 결정론적 / 반복 워크플로 양쪽에서 활용 가능"

### 7. 참고 자료 URL 리다이렉트

- **위치**: 모든 패턴 파일의 참고 자료 섹션
- **현재**: `https://cloud.google.com/architecture/choose-design-pattern-agentic-ai-system`
- **문제**: 해당 URL은 현재 `https://docs.cloud.google.com/architecture/choose-design-pattern-agentic-ai-system`으로 301 리다이렉트됨. ADK 링크(`google.github.io/adk-docs/`)도 모두 루트 페이지를 가리키며 패턴별 페이지로 연결되지 않음.
- **수정 제안**: 현재 정규 URL(`docs.cloud.google.com`)로 업데이트. ADK 링크는 가능한 경우 패턴별 페이지로 변경하거나, 루트 링크임을 명시.

### 8. README의 ReAct 패턴 분류 위치

- **위치**: `README.md` 30-34행 "추가 패턴" 섹션
- **현재**: ReAct가 Human-in-the-Loop, 맞춤 로직과 함께 "추가 패턴"으로 분류
- **문제**: Google Cloud 원문에서 ReAct는 단일/멀티 에이전트와 별개의 독립 섹션으로 설명되며, 패턴 선택 표에서는 "반복이 포함된 워크플로"에 분류. "추가 패턴"이라는 명칭은 원문에 없는 임의 분류.
- **원문 근거**: 원문의 패턴 선택 가이드에서 ReAct는 "반복이 포함된 워크플로" 카테고리에 배치. Human-in-the-Loop와 맞춤 로직은 "특별한 요구사항이 있는 워크플로"에 배치. 세 패턴이 동일 카테고리가 아님.
- **수정 제안**: "추가 패턴" 대신 원문의 분류를 따라 ReAct를 별도 구분하거나, "반복 포함 워크플로"에 편입하고 HITL/맞춤 로직은 "특수 요구사항 패턴"으로 분리

## Suggestions

### 9. Human-in-the-Loop의 교차 패턴 특성 명시

- **위치**: `11-human-in-the-loop.md`
- **현재**: 독립 패턴으로만 설명
- **제안**: Human-in-the-Loop는 다른 패턴(순차, 코디네이터 등)의 특정 단계에 삽입할 수 있는 교차 적용 패턴임을 개요에 명시. 예: "이 패턴은 독립적으로 사용될 수 있으며, 다른 패턴(순차, 코디네이터 등)의 특정 단계에 체크포인트로 삽입하여 결합할 수도 있습니다."

### 10. 검토-비평 패턴의 ADK 참고 자료 링크 텍스트

- **위치**: `05-review-critique.md` 참고 자료 섹션
- **현재**: 링크 텍스트가 "Google ADK: Loop Agents"
- **제안**: 검토-비평 패턴이 루프 패턴의 특수한 형태임을 링크 텍스트에서 명시. 예: "Google ADK: Loop Agents (검토-비평 패턴의 기반)"

### 11. 코디네이터 패턴 비교표의 병렬 실행 가능성

- **위치**: `07-coordinator.md` 병렬 패턴과의 차이 비교표
- **현재**: 코디네이터의 에이전트 실행이 "필요한 에이전트만 선택적 실행"으로 표현
- **제안**: 코디네이터도 복수의 에이전트를 동시에 디스패치할 수 있음을 반영. "필요한 에이전트만 선택적/동시 실행 (AI 모델이 결정)"으로 수정하면 더 정확.

### 12. 반복 개선 vs 검토-비평 비교표 보완

- **위치**: `06-iterative-refinement.md` 비교표의 "상태 관리" 행
- **현재**: 검토-비평은 "출력 전달", 반복 개선은 "세션 상태에 저장/수정"
- **제안**: 원문에서 두 패턴 모두 루프 패턴의 구현체이며 기술적 상태 관리 차이보다는 초점(검증 vs 개선)의 차이가 본질적. 비교표에 "두 패턴 모두 세션 상태를 활용할 수 있으나, 검토-비평은 승인/거부에, 반복 개선은 결과물 버전 관리에 초점"이라는 보충 설명 추가 권장.

## Positive Findings

- **일관된 문서 구조**: 12개 문서 모두 동일한 템플릿 (개요 → 아키텍처 → 사용 예시 → 장단점 → 참고 자료) 준수
- **Mermaid 다이어그램 품질**: 27개 다이어그램 모두 유효한 문법. 아키텍처(flowchart) + 동적 흐름(sequenceDiagram) 이중 구성이 정적 구조와 동적 행위를 효과적으로 전달
- **원문에 없는 비교표 추가**: 06(반복 개선 vs 검토-비평), 07(코디네이터 vs 병렬), 08(계층적 vs 코디네이터), 09(스웜 vs 코디네이터) 문서에 관련 패턴 간 비교표를 추가하여 원문 대비 독자 가치 향상
- **실용적 보충 섹션**: 루프 패턴의 "위험 관리"(04), HITL의 "설계 고려사항"(11), 맞춤 로직의 "설계 고려사항"(12) 등 원문에 없는 실무 지침 포함
- **한국어 품질**: 전문 용어 일관성(에이전트, 워크플로, 오케스트레이션 등), 자연스러운 문체, 문법 오류 없음
- **README 결정 flowchart**: 원문의 4개 표 형식 선택 가이드를 하나의 시각적 의사결정 트리로 통합하여 사용성 향상
- **내부 링크 무결성**: README의 12개 패턴 링크가 모두 실제 파일명과 일치하며 깨진 링크 없음
