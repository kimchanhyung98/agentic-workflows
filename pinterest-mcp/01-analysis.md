# Pinterest MCP 설계 및 워크플로우 분석

## 1. 개요

Pinterest는 에이전트 활용을 확장하면서, 내부 도구 연결 방식을 MCP(Model Context Protocol) 중심으로 표준화하는 방향을 제시했다.
핵심은 "각 팀이 도구를 제각각 붙이는 구조"에서 "플랫폼 레벨에서 재사용 가능한 MCP 생태계"로 전환하는 것이다.

> 참고: 본 문서는 제공된 참고 링크와 MCP 공식 문서를 기반으로 정리했다.
> 현재 실행 환경 제약으로 원문을 직접 재수집하지 못한 항목은 참고 링크 기준으로 요약했다.

## 2. Pinterest의 AI 사용/활용 방식(요약)

공개 내용을 종합하면 Pinterest의 방향은 다음과 같이 정리된다.

### 2.1 도구 연결의 표준화

- 에이전트가 사용하는 내부 시스템(코드, 데이터, 운영 도구)을 **MCP 인터페이스로 표준화**
- 에이전트/모델이 바뀌어도 도구 계층을 재사용할 수 있게 설계
- 팀 단위 실험을 플랫폼 차원으로 확장하기 위한 공통 기반 확보

### 2.2 생태계 접근(단일 서버가 아닌 다중 서버)

- MCP를 단일 기능 서버가 아니라 **여러 도메인 서버의 생태계**로 운영
- 레지스트리/디렉터리 개념을 두어 서버 발견성과 재사용성을 높임
- 신규 MCP 서버를 점진적으로 추가해 유즈케이스를 넓히는 방식

### 2.3 운영 중심의 엔지니어링

- 단순 연결보다 **권한, 정책, 관측성, 품질 평가**를 함께 운영
- 도구 호출 실패, 지연, 품질 편차를 측정하고 서버/스키마를 개선
- 결과적으로 AI 기능을 "데모"가 아닌 "내부 플랫폼"으로 관리

## 3. MCP 작동 방식 분석

Pinterest 사례를 MCP 표준 관점으로 해석하면 기본 메커니즘은 다음과 같다.

### 3.1 런타임 상호작용

1. Agent Host(IDE/CLI/Chat)가 사용자 요청을 전달
2. 에이전트가 MCP 서버 목록과 도구 스키마를 조회
3. 필요한 도구를 선택해 MCP(JSON-RPC) 호출
4. MCP 서버가 내부 시스템/API를 호출해 결과 반환
5. 에이전트가 결과를 합성해 사용자에게 응답

### 3.2 왜 MCP가 유리한가

- **호환성**: 모델/프레임워크가 바뀌어도 도구 계약 유지
- **재사용성**: 도메인 서버를 여러 팀과 에이전트가 공유 가능
- **거버넌스**: 인증/인가/감사를 공통 계층에서 다루기 쉬움
- **확장성**: 신규 도구를 서버 단위로 안전하게 추가 가능

## 4. Pinterest MCP 워크플로우(운영 관점)

Pinterest의 "생태계" 포인트를 워크플로우로 재구성하면 아래와 같다.

### 4.1 서버 온보딩

- 신규 도구/시스템을 MCP 서버로 래핑
- 도구 스키마, 설명, 접근 정책, 소유팀 메타데이터 등록
- 테스트 및 샌드박스 검증 후 레지스트리에 공개

### 4.2 실행 경로

- 요청 접수 → 도구 후보 탐색 → 정책 검증 → 서버 호출 → 응답 합성
- 실패 시 재시도/대체 도구/안전한 실패 메시지 전략 적용
- 실행 데이터는 관측성 시스템에 기록

### 4.3 피드백 루프

- 호출 로그 기반으로 실패 패턴/미사용 도구/지연 구간 분석
- 스키마 개선, 프롬프트 힌트 보강, 정책 조정, 서버 성능 튜닝
- 개선 결과를 재배포하여 성공률과 사용자 경험을 지속적으로 향상

## 5. 설계 시사점

Pinterest 사례는 MCP를 "프로토콜 도입"으로 끝내지 않고, 플랫폼 운영 체계까지 포함해 확장한 점이 핵심이다.

- **프로토콜 + 플랫폼** 동시 설계가 중요
- **도메인 소유팀 + 중앙 플랫폼팀**의 역할 분리가 필요
- **관측/평가 지표** 없이는 생태계 품질 유지가 어려움
- **정책/보안 내재화**가 없으면 엔터프라이즈 확장에 한계
