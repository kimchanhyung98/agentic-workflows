# Pinterest MCP 설계 및 운영 분석

## 1. 개요

- 출처: Pinterest Engineering 블로그 (2026년 3월)
- 성격: 엔터프라이즈 규모 MCP 생태계 구축 사례
- 핵심: 도메인별 소형 MCP 서버 + 중앙 레지스트리 + 2층 보안 모델 + 통합 배포 파이프라인
- 주요 성과: 월 66,000+ 호출, 844 MAU, ~7,000시간/월 절감

Pinterest는 "MCP가 흥미해 보인다"는 시작점에서, 다수의 도메인별 MCP 서버, 중앙 레지스트리, IDE·사내 챗·AI 에이전트에 걸친 프로덕션 통합을 운영하는 단계까지 도달했다.

---

## 2. 도입 배경

### 2.1 문제 인식

각 팀이 에이전트에 사내 도구를 제각각 연결하는 구조에서, 다음 문제가 반복됐다.

- 모델/프레임워크가 바뀔 때마다 도구 연동을 다시 구현
- 팀마다 인증·권한·로깅 방식이 달라 거버넌스가 어려움
- 도구 발견성이 낮아 중복 구현이 발생

### 2.2 목표

"각 팀이 도구를 제각각 붙이는 구조"에서 "플랫폼 레벨에서 재사용 가능한 MCP 생태계"로 전환하는 것이다.

- 에이전트가 사내 도구를 일관된 인터페이스(MCP)로 호출하도록 표준화
- 모델/에이전트가 바뀌어도 도구 계층을 재사용
- 서버 등록/발견, 권한, 관측성, 품질 평가를 플랫폼 관점에서 통합

---

## 3. 핵심 설계 결정

### 3.1 도메인별 소형 서버 전략

Pinterest는 하나의 거대한 MCP 서버가 아니라, **도메인별로 작은 서버를 여러 개 운영**하는 방식을 택했다.

이유:

- **접근 제어 분리**: 서버마다 다른 보안 정책을 적용할 수 있다 (예: Presto 서버는 business group 기반 접근 제한)
- **모델 컨텍스트 오염 방지**: 서버당 소수의 관련 도구만 노출해, 에이전트가 불필요한 도구에 혼동되지 않게 한다

### 3.2 "코딩 질문"이 아닌 "엔지니어링 자동화"

Pinterest는 MCP를 단순한 AI 코딩 보조가 아니라, **엔지니어링 작업을 안전하게 자동화하는 기반**으로 접근했다.
에이전트가 데이터 조회, 장애 진단, 워크플로우 관리를 직접 수행하되, human-in-the-loop 원칙으로 통제한다.

## 4. MCP 서버 상세

### 4.1 Presto MCP

가장 높은 트래픽을 기록하는 서버다.
에이전트(IDE 포함)가 Presto 기반 데이터를 직접 조회해, 대시보드로 컨텍스트 전환할 필요 없이 워크플로우 안에서 데이터를 사용할 수 있게 한다.

- business group 기반 접근 제어 적용 (예: Ads-eng만 `get_revenue_metrics` 호출 가능)
- 일부 도구는 JWT 기반 사용자 수준 attribution 필수

### 4.2 Spark MCP

AI Spark 디버깅 경험의 핵심이다.
Spark 작업 실패를 진단하고, 로그를 요약하며, 구조화된 근본 원인 분석(root-cause analysis)을 기록하는 데 사용된다.

### 4.3 Knowledge MCP

범용 지식 엔드포인트다.
사내 AI 봇이 회사 지식과 Q&A에 사용하고, 다른 에이전트들이 내부 문서와 디버깅 질문에 대한 답을 찾는 데 활용한다.

### 4.4 Airflow MCP

워크플로우 오케스트레이션 도구와의 연동을 담당한다.

## 5. MCP Registry

MCP Registry는 **승인된 MCP 서버의 단일 source of truth**다.

### 5.1 Web UI (사람용)

- 서버 발견: 어떤 서버가 있는지, 소유팀은 누구인지, 지원 채널은 어디인지
- 보안 포스처: 각 서버의 보안 수준 확인
- 실시간 상태: 서버의 live status와 노출 도구 목록

### 5.2 API (AI 클라이언트용)

- 서버 목록 조회와 검증
- "이 사용자가 서버 X를 사용할 수 있는가?" 확인
- 내부 서비스가 에이전트의 서버 접근을 사전 검증하는 데 사용

### 5.3 거버넌스

Registry에 등록된 서버만 "프로덕션 사용 승인" 상태로 인정된다.
미등록 서버는 프로덕션 에이전트에서 호출할 수 없다.

## 6. 보안 모델

Pinterest MCP 생태계의 보안은 **2층 인가 모델**로 구성된다.

### 6.1 1층: 네트워크/서비스 수준 (Envoy)

클라이언트가 내부 OAuth 스택에 대해 인증하고, 결과 JWT를 MCP Registry와 대상 MCP 서버에 전송한다.

**Envoy 프록시가 수행하는 작업:**

- JWT 검증
- `X-Forwarded-User`, `X-Forwarded-Groups` 등 헤더 매핑
- Coarse-grained 보안 정책 적용
  - 예: "프로덕션의 AI chat webapp은 Presto MCP 서버에 접근 가능하지만, dev 네임스페이스의 실험 서버에는 접근 불가"

### 6.2 2층: 도구 수준 (서버 내부)

서버 내부에서 `@authorize_tool(policy='...')` 데코레이터로 fine-grained 규칙을 적용한다.

- 예: Ads-eng 그룹만 `get_revenue_metrics`를 호출할 수 있다 (서버 자체에는 다른 조직도 접근 가능하더라도)
- 일부 서버는 도구 발견(discovery) 단계에서도 유효한 JWT를 요구해, 모든 호출에 대한 사용자 수준 attribution을 보장한다

### 6.3 대체 경로: SPIFFE 기반 서비스 간 인증

저위험, 읽기 전용 시나리오에서는 SPIFFE 기반 mesh identity만으로 인증할 수 있다.

- 내부 서비스 메시가 보안 정책을 강제하고, 서버는 호출 서비스의 mesh identity로 인가를 판단한다
- 최종 사용자가 관여하지 않고, blast radius가 좁은 경우에만 사용한다

### 6.4 Human-in-the-Loop

MCP 서버는 자동화된 작업을 수행하므로, 사람이 직접 도구를 사용하는 것보다 blast radius가 크다.
Pinterest의 에이전트 가이드라인은 **민감하거나 비용이 큰 작업은 반드시 사람의 승인을 받도록** 명시하고 있다.

## 7. 배포 인프라

### 7.1 초기 문제

초기에 가장 많이 들어온 피드백은 "새 MCP 서버를 만드는 데 너무 많은 작업이 필요하다"였다.

### 7.2 통합 배포 파이프라인

이를 해결하기 위해 Pinterest는 **모든 MCP 서버의 인프라를 일괄 처리하는 통합 배포 파이프라인**을 만들었다.

- 도메인 팀은 도구 로직(비즈니스 로직)만 정의한다
- 플랫폼이 배포, 스케일링, 서비스 인프라를 자동으로 처리한다
- 도메인 전문가가 배포 메커니즘이 아니라 비즈니스 로직에 집중할 수 있게 한다

## 8. 에이전트 통합 표면

Pinterest MCP 서버는 다음 세 가지 표면에서 사용된다.

| 표면 | 설명 |
|---|---|
| AI IDE | Cursor 등 AI 지원 IDE에서 에이전트가 MCP 도구를 직접 호출 |
| 사내 AI 챗 | 내부 챗 인터페이스에서 Knowledge MCP 등을 활용한 Q&A |
| AI 에이전트 | 자동화된 엔지니어링 작업을 수행하는 독립 에이전트 |

---

## 9. 운영 지표와 피드백 루프

### 9.1 현재 운영 규모

| 항목 | 수치 |
|---|---|
| 월간 MCP 호출 수 | 66,000+ |
| 월간 활성 사용자 (MAU) | 844명 |
| 월간 절감 시간 (추정) | ~7,000시간 |

### 9.2 피드백 루프

1. 호출 로그와 사용량 데이터를 수집한다
2. 실패 패턴, 미사용 도구, 지연 구간을 분석한다
3. 스키마 개선, 정책 조정, 서버 추가/성능 튜닝을 수행한다
4. 통합 배포 파이프라인을 통해 업데이트를 반영한다

## 10. 설계 시사점

Pinterest 사례에서 추출할 수 있는 시사점은 다음과 같다.

| 시사점 | 설명 |
|---|---|
| 소형 서버 분리 | 하나의 큰 서버보다 도메인별 소형 서버가 접근 제어와 컨텍스트 관리에 유리하다 |
| 레지스트리 = 거버넌스 | 승인/발견/상태 모니터링을 하나의 레지스트리에 집중시켜 거버넌스의 기반으로 삼는다 |
| 2층 보안 | 네트워크 수준(Envoy) + 도구 수준(데코레이터)으로 coarse/fine-grained 정책을 분리한다 |
| 배포 부담 제거 | 도메인 팀이 인프라를 몰라도 MCP 서버를 만들 수 있어야 생태계가 성장한다 |
| 관측성 → 개선 | 호출 로그 기반 피드백 루프 없이는 생태계 품질을 유지할 수 없다 |
| human-in-the-loop | 에이전트의 자동화 범위가 넓어질수록, 민감 작업에 대한 사람 승인이 필수다 |
